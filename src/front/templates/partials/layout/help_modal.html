Starter Guide — Build a Digital Twin in 4 Clicks

OntoBricks can generate an end-to-end knowledge graph from your Databricks tables using LLM-powered automation. Here is the fastest path from zero to a queryable graph.

Prerequisites
  • A Databricks workspace with Unity Catalog enabled
  • A running SQL Warehouse
  • A UC Volume for the domain registry (configure via Settings → Registry → Initialize if empty)
  • A Databricks Model Serving endpoint (for LLM-powered automation)
1
Create a Domain

Open Registry → click New Domain. Give it a name (e.g. sales_demo), then open the Domain tab and click Save Domain (Cmd/Ctrl+S). The domain must be saved before ontology, mapping and digital-twin features unlock.

2
Import Metadata

Go to Domain → Data Sources, pick your catalog and schema, then click Import Metadata. OntoBricks fetches table and column metadata from Unity Catalog.

Import Documents Optional

Before generating, open Domain → Documents and attach any reference material (PDFs, schemas, data dictionaries, glossaries, …). The LLM uses these documents as additional context when producing the ontology — improving entity names, descriptions, and relationships.

3
Generate the Ontology

Open Ontology → Generate (AI Wizard), select your LLM endpoint, optionally choose a Quick Template (CRM, IoT, Healthcare, Energy, ...) and click Generate. The LLM produces entities, relationships and attributes — grounded in the metadata and any documents you imported. Review in the Model tab.

4
Map your data & Sync

Open Mapping to connect your ontology to Databricks tables. You can either map manually using the Designer or Manual views (click each entity / relationship, write or tweak the SQL, assign columns), or let an agent do it for you via Auto-Map → Start Auto-Map. Once mappings are in place, go to Digital Twin → Sync and click Synchronize — triples are materialized into your triple store.

Explore

Head to Digital Twin → Knowledge Graph to browse your data as an interactive graph, or use GraphQL for programmatic access.

Tip: You can navigate away from long-running tasks (Auto-Map, Sync, Data Quality) — they run in the background and their progress is tracked in the Background Tasks panel in the navbar.

Step-by-Step Workflow

The full OntoBricks workflow, in detail. Each step corresponds to a main navbar item.

The Registry is the central index of all domains (ontology + mapping + triple store) stored in your Unity Catalog Volume.

  • Browse — list and open existing domains
  • New Domain (navbar) — create a fresh domain
  • Load Domain (navbar) — open an existing domain from UC
  • Bridges — cross-domain links that allow navigating from one knowledge graph to another
  • Schedule — automation / recurring sync jobs
  • API — REST API reference for developers
  • Registry Location — UC Volume where domains are persisted

  • Information — name, description, base URI, custom URI override
  • Versions — manage snapshots of your domain
  • Cockpit — overall readiness dashboard (ontology ✓, mapping ✓, sync ✓)
  • Data Sources — import Unity Catalog metadata (catalog, schema, tables, columns)
  • Documents — attach PDFs, schemas and other references for context

Save your domain (Cmd/Ctrl+S) before moving to Ontology or Mapping.

Four ways to build an ontology:

  1. Model (Designer) — drag & drop entities, relationships, inheritance arrows on the OntoViz canvas. Auto-save. Click the smiley face () to auto-assign emoji icons via LLM.
  2. Generate (Wizard) — AI generates the full ontology from your UC metadata. Pick a Quick Template or write custom guidelines.
  3. Import — load OWL/RDFS files, or pull industry standards: FIBO, CDISC, IOF.
  4. Form-based — use the Entities and Relationships sidebar sections to add classes and object properties manually.

Advanced features (Advanced section):

  • Data Quality (SHACL) — define shapes for completeness, cardinality, pattern, consistency...
  • Business Rules (SWRL) — author inference rules using the graphical D3 editor
  • Expr. & Axioms — equivalent, disjoint, union, inverse, property chains...
  • Groups — organize entities logically for readability
  • Business Views — curated subsets for end-users

Preview the generated OWL/Turtle under W3C Standards → OWL.

Several complementary interfaces:

  • Designer — force-directed graph of your ontology. Click nodes to open the mapping panel (Wizard / SQL / Mapping tabs).
  • Manual — tree grouped by mapping status (unmapped, partial, mapped)
  • Auto-Map — LLM generates SQL mappings for every entity and relationship in one shot
  • Import — load mappings from a file
  • Diagnostics — consistency checks on the mapping
  • R2RML / Spark SQL — preview generated code

In the mapping panel, the Mapping tab lets you assign columns as ID, Label or a specific attribute, with a live data preview.

Mappings store the query without a LIMIT — the preview limit only affects the UI.

  • Overview — insight dashboard: entity counts, relationship counts, last sync timestamp, quality status
  • Knowledge Graph — interactive sigma.js WebGL graph. Search, filter, detect clusters (Louvain / Label Propagation / Greedy Modularity), color-by-cluster, collapse into super-nodes
  • GraphQL — auto-generated GraphQL playground for programmatic access
  • Data Quality — run SHACL / constraint validation against the triple store
  • Inference — OWL 2 RL & SWRL reasoning on Delta / Lakebase (SQL)

A Readiness Status indicator tells you when ontology & mapping are both ready to sync.

Feature Guide

AI Assistant

A floating assistant in the Model view lets you edit the ontology with natural-language commands (“add entity Customer”, “remove orphans”, “list relationships”...).

Interactive Knowledge Graph

Two-phase search, configurable depth, cross-domain bridges, cluster detection (client-side Graphology / server-side NetworkX), color-by-cluster, super-nodes.

Dashboard embedding

Assign a Databricks dashboard to an entity type; parameters are auto-populated from entity attributes when viewed in the Digital Twin.

SHACL Data Quality

Six categories (Completeness, Cardinality, Uniqueness, Consistency, Conformance, Structural) executed as SQL or PySHACL. Per-rule progress, violation caps, severity levels.

SWRL Reasoning

Graphical D3 rule editor. Rules compile to SQL (Delta or Lakebase). Per-rule progress tracking during inference.

Cross-domain Bridges

Link entities from different domains. The navbar bridges navigation automatically switches domains and focuses the target entity.

MCP Integration

Expose the knowledge graph to LLM agents through the Model Context Protocol. Deploy the companion mcp-ontobricks app and connect from Cursor, Claude Desktop or the Databricks Playground.

Unity Catalog first

All domains are stored in UC volumes; triples land in Delta tables and the configured graph backend (Lakebase). Incremental sync by default.

Troubleshooting

“The warehouse icon is grey/yellow.”

Your SQL Warehouse is not yet configured or is starting. Click the icon (admin-only) or go to Settings to pick a warehouse and wait for it to reach the running state.

“Registry is empty / no domains shown.”

Open Settings → Registry and click Initialize. This creates the required folder structure in your Unity Catalog Volume.

“Auto-Map failed on some entities.”

Open Mapping → Auto-Map — the report table shows which entities failed and why. For entities with missing attribute mappings only, click Re-Assign Missing Attributes to retry just those.

“Sync is very slow.”

Large mappings benefit from a larger SQL Warehouse. Also check that your mappings are reasonable SQL (avoid unnecessary cross-joins). Data movement is handled by Databricks Lakeflow synced tables.

“Session state leaked after switching domains.”

Refresh the browser once. The latest version resets all per-domain state on switch — if you still see staleness, report it with a changelog reference.

“I see ‘requires domain saved’ tooltips everywhere.”

You must first create and save a domain. Registry → New Domain → then on the Domain page press Cmd/Ctrl+S.

Glossary

Ontology
Formal model of entities (classes), their attributes (data properties) and relationships (object properties), expressed in OWL.
Entity / Class
A concept in the domain (e.g. Customer, Order).
Attribute / Data Property
A scalar value attached to an entity (e.g. email, birthDate).
Relationship / Object Property
A link between two entities (e.g. Customer placedOrder Order).
OWL
W3C Web Ontology Language — the vocabulary OntoBricks uses to describe the ontology.
RDF / Triple
Subject-Predicate-Object statement — the native data model of the triple store.
R2RML
W3C standard for mapping relational data (SQL tables) to RDF triples.
SHACL
W3C Shapes Constraint Language — used for data quality validation.
SWRL
Semantic Web Rule Language — “IF ... THEN ...” inference rules.
Digital Twin
A materialized, queryable knowledge graph representation of your Databricks data.
Bridge
Cross-domain link between entities of two different domains.
Triple Store
Backend where triples are physically stored (Delta tables in Unity Catalog).
Graph DB
Engine that exposes the triples for graph queries and reasoning (currently Lakebase Postgres).
Unity Catalog (UC)
Databricks unified governance for catalogs, schemas, tables and volumes.
MCP
Model Context Protocol — exposes the knowledge graph to LLM agents.

Domain

List registry domains, discover versions, inspect design readiness (ontology, data sources, mappings), and export artifacts (OWL, R2RML, generated Spark SQL). Optional registry overrides: registry_catalog, registry_schema, registry_volume.

GET /api/v1/domain/r2rml Get R2RML mapping

Returns the domain's R2RML mapping document in Turtle format. R2RML defines how relational tables map to RDF triples — each TriplesMap specifies a source table, subject template, and predicate-object mappings.

Query Parameters
ParameterTypeDefaultDescription
domain_namestring—Domain name in the registry (uses current session if omitted)
domain_versionstring—Specific version to load (uses latest if omitted)
Sample Response
{
  "success": true,
  "format": "turtle",
  "content": "@prefix rr: <http://www.w3.org/ns/r2rml#> .\n\n:CustomerMap a rr:TriplesMap ;\n    rr:logicalTable [ rr:sqlQuery \"SELECT * FROM customers\" ] ;\n    ...",
  "base_uri": "https://ontobricks.com/ontology/",
  "entity_count": 5,
  "relationship_count": 4
}
GET /api/v1/domain/sparksql Get generated Spark SQL

Returns the Spark SQL query generated from the domain's R2RML mappings. This is the SQL that produces (subject, predicate, object) triples when executed against the source tables. The dialect adapts to the triple store backend (Spark SQL for Delta).

Query Parameters
ParameterTypeDefaultDescription
domain_namestring—Domain name in the registry (uses current session if omitted)
domain_versionstring—Specific version to load (uses latest if omitted)
Sample Response
{
  "success": true,
  "sql": "SELECT subject, predicate, object FROM (\n  SELECT\n    CONCAT('https://ontobricks.com/ontology/Customer/', customer_id) AS subject,\n    'http://www.w3.org/1999/02/22-rdf-syntax-ns#type' AS predicate,\n    'https://ontobricks.com/ontology/Customer' AS object\n  FROM customers\n  UNION ALL\n  ...\n)",
  "base_uri": "https://ontobricks.com/ontology/",
  "dialect": "spark"
}

GET /api/v1/digitaltwin/triples/find Find & traverse

Search for entities by type and/or label text, then traverse their relationships up to N levels deep (BFS graph walk). Returns all triples discovered during traversal, similar to the Knowledge Graph's Find & Filter feature.

Query Parameters
ParameterTypeDefaultDescription
domain_namestring—Domain name in the registry (uses current session if omitted)
domain_versionstring—Specific version to load (uses latest if omitted)
entity_typestring—Filter by rdf:type local name (exact, case-insensitive)
searchstring—Text search on entity labels (rdfs:label) or URI local name
depthinteger1Traversal depth (1 = direct neighbors, max 10)
limitinteger1000Max triples per page (1–10 000)
offsetinteger0Number of triples to skip (for pagination)
Sample Response
{
  "success": true,
  "seed_count": 3,
  "depth": 2,

POST /api/v1/digitaltwin/inference/start Run inference

Start OWL 2 RL inference, SWRL rule execution, graph reasoning, and constraint checking as an asynchronous task. Each phase can be toggled independently. Optionally append inferred triples to the knowledge graph or materialise to a Delta table. Returns a task_id to poll progress via GET /inference/{task_id}.

Query Parameters
ParameterTypeDefaultDescription
domain_namestring—Domain name in the registry (uses current session if omitted)
domain_versionstring—Specific version to load (uses latest if omitted)
Request Body
FieldTypeDefaultDescription
tboxbooleantrueRun T-Box (OWL 2 RL) inference
swrlbooleantrueRun SWRL rule execution
graphbooleantrueRun graph-structural reasoning
constraintsbooleantrueRun constraint checks
append_graphbooleanfalseAppend inferred triples to the knowledge graph
materializebooleanfalseWrite inferred triples to a Delta table
materialize_tablestringnullFully-qualified table name (catalog.schema.table)
Sample Request
POST /api/v1/digitaltwin/inference/start?domain_name=my_domain
Content-Type: application/json

{
  "tbox": true,
  "swrl": true,
  "graph": true,
  "constraints": true,
  "append_graph": false,
  "materialize": false
}
Sample Response
{
  "success": true,
  "task_id": "rs-xyz789",
  "message": "Inference started (phases: tbox, swrl, graph, constraints)"
}

Cohort

List saved cohort rules, dry-run a candidate rule against the active graph, and materialise a saved rule to graph triples and/or a Unity Catalog Delta table.

OpenAPI: Cohort

POST /api/v1/digitaltwin/cohorts/materialize Materialise a saved rule

Re-run a saved cohort rule and persist the outputs as configured by the rule (graph triples, UC Delta table). Use output_graph / output_uc to skip a target for this run only — the saved rule is unchanged. Materialisation is idempotent: re-running replaces previous output for the same rule.

Query Parameters
ParameterTypeDefaultDescription
domain_namestring—Domain name in the registry (uses current session if omitted)
domain_versionstring—Specific version to load (uses latest if omitted)
Request Body
FieldTypeDefaultDescription
rule_idstringrequiredId of the saved cohort rule to run
output_graphbooleannullOverride the rule's output.graph for this run only (null honours the saved rule)
output_ucbooleannullOverride the rule's UC-table output for this run only
Sample Request
POST /api/v1/digitaltwin/cohorts/materialize?domain_name=consulting
Content-Type: application/json

{ "rule_id": "Staffing001", "output_uc": false }
Sample Response
{
  "success": true,
  "rule_id": "Staffing001",
  "cohort_count": 4,
  "grouped_member_count": 12,
  "elapsed_ms": 132,
  "materialized_triples": 28,
  "uc_rows_written": 0,
  "uc_table": null
}
All endpoints accept a domain_name query parameter to select a domain from the registry. If omitted, the current session's domain is used. When called from external clients, ensure the Databricks workspace URL and token are configured.

GraphQL API

GET /api/v1/graphql List GraphQL-enabled domains

Returns domains that are marked Active and have a populated ontology — the prerequisites for auto-generating a GraphQL schema.

Query Parameters

None

Sample Response

  "domains": [

}

GET /api/v1/graphql/{domain_name}/schema Schema introspection (SDL)

Returns the auto-generated GraphQL schema in SDL (Schema Definition Language) format. The schema reflects the domain's ontology: each class is a type, each data property a field, and each object property a typed relationship.

Path Parameters
ParameterTypeDescription
domain_namestringDomain name in the registry
Sample Response
{
  "sdl": "type Customer {\n  id: ID!\n  uri: String!\n  label: String\n  firstname: String\n  lastname: String\n  email: String\n  hasInteraction: [Interaction!]\n}\n\ntype Interaction {\n  id: ID!\n  uri: String!\n  label: String\n  date: String\n}\n\ntype Query {\n  customers(limit: Int! = 50, offset: Int! = 0, search: String = null): [Customer!]!\n  customer(id: ID!): Customer\n  interactions(limit: Int! = 50, offset: Int! = 0, search: String = null): [Interaction!]!\n  interaction(id: ID!): Interaction\n}"
}
The GraphQL schema is auto-generated from the domain's ontology. Change the ontology and the schema updates automatically — no hand-written types needed. Only domains marked Active with a populated ontology expose a GraphQL endpoint.
